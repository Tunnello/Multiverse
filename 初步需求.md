# 🚀 产品需求文档 (PRD)：知乎“观点对撞机” (Zhihu Multiverse)

**文档版本：** V1.0 (Hackathon 冲刺版)
**日期：** 2026年5月4日
**产品定位：** 基于大模型的复杂议题可视化探索引擎

## 1. 项目概述

**1.1 背景与痛点：**
知乎沉淀了大量高质量的长图文问答，但在面对争议性大、跨度长的宏大议题时，用户往往面临“信息过载”。阅读单篇长文耗时过高，且难以快速看清整个议题在不同立场、不同时间点下的“认知全貌”与“观点分歧”。

**1.2 核心目标：**
将传统的“线性阅读摘要”升级为“空间化、时空化的动态图谱探索”。通过 AI 提取长文逻辑提取并结构化输出，构建出直观的“知识星团”，让用户一秒看懂争议内核与共识演化。

## 2. 核心价值与业务闭环 (评委视角)

* **内容盘活：** 将知乎沉睡的“历史高质量长回答”作为核心资产，赋予其二次曝光和交互的价值。
* **降低门槛：** 把硬核的图文阅读转化为可视化的“解密探索”，提升年轻用户和轻度用户的停留时长。
* **极致契合 IP：** 完美映射知乎“有问题，就会有答案（与争议）”的社区灵魂，极具视觉表现力。

## 3. 功能模块与交互需求 (P0级核心链路)

### 3.1 AI 提炼与结构化引擎 (后端/AI)

针对特定的知乎问题（如“新能源车真的省钱吗？”），大模型需要抓取 Top 20 的高赞回答，并输出标准化的 JSON 节点数据。

* **立场提取：** 总结答主的核心论点。
* **流派分类：** 强制将答主归类至四大预设阵营（用于前端颜色编码）：

  * 学术派/专业探讨 (蓝色)
  * 激进派/批判颠覆 (红色)
  * 经验派/个人实操 (绿色)
  * 利益相关/从业者 (紫色)
* **关系推演：** 找出各个回答之间的逻辑关系（赞同/互斥/补充），并生成边缘连线 (Edges) 的解释词。

### 3.2 观点对撞可视化大屏 (前端)

摒弃传统列表，采用全屏的力导向图 (Force-Directed Graph) 进行渲染。

* **星团聚类 (Clustering)：** 立场相似的节点受引力影响聚合成团。
* **对撞连线 (Colliding)：** 立场相左的星团之间，绘制带有动效的红色虚线（表示冲突），并在连线上悬浮展示“争议焦点”。
* **节点交互：** 点击单颗星球（节点），右侧滑出抽屉，展示该回答的极简摘要、核心金句与作者标签。

### 3.3 核心亮点：时空漂移 (Time Drift) 控件

展示议题在时间维度上的认知演变。

* **时间轴控制：** 界面底部配置年份滑动条（如 2016 —— 2026）。
* **动态渲染：** 随着滑块拖动，图谱动态更新。早年尚未发布的回答节点处于隐藏状态；用户可以直观看到观点是如何随着时间推移，从“极度分散对立”最终走向“某个维度的共识”。
* **现实复检打分：** 对比历史预测与 2026 年的客观现实，在右侧抽屉面板给予该回答一个“预测偏离值”评分。

## 4. 技术栈选型建议

| 模块        | 推荐技术/工具              | 选型理由 (Hackathon 导向)                   |
| :-------- | :------------------- | :------------------------------------ |
| **前端可视化** | AntV G6 / ECharts    | 开箱即用，内置成熟的力导向图和聚类动效，降低手写 Canvas 的风险。  |
| **前端框架**  | React 或 Vue 3        | 团队最熟悉的框架，保证开发速度。                      |
| **后端服务**  | Python (FastAPI)     | 代码轻量，与大模型 API 对接最顺滑，数据处理灵活。           |
| **AI 模型** | 主办方指定 API (或 GPT-4o) | 强调强大的指令遵循能力，确保输出稳定的 JSON 格式，避免数据解析崩溃。 |

## 5. 风险预案 (Plan B)

**风险点：** 现场演示时，网络拥堵或大模型 API 响应超时，导致图谱渲染白屏。
**应对策略 (降级预案)：** 实施“双轨制”。开发阶段利用真实 API 跑通核心链路。但在 Demo 演示版本中，将精选的 2-3 个黄金命题的 AI 解析结果（JSON 数据）**固化到本地**。前端优先加载本地预设数据，配合 Loading 动画，确保评委体验到绝对流畅、零延迟的视觉交互和对撞效果。
